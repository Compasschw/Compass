# Compass CHW — Pre-Launch Audit

**Date:** 2026-06-12
**Scope:** 5-dimension audit prepared by 5 parallel specialist agents (security, performance, UI/UX, production readiness, code architecture). Synthesized by Opus 4.7 (1M context). The lens throughout is "what blocks a real Medi-Cal member from a safe, smooth first session?"
**Overall posture:** **Strong foundations, several specific blockers, but not far from launch-ready.** None of the findings require a re-architecture. Most blockers are 5-minute to 4-hour fixes.

## Verdict by dimension

| Dimension | Result | Confidence-to-ship |
|---|---|---|
| Security + HIPAA | **5 blockers / 10 concerns** — fundamentals strong (BAA gates, encryption, relationship guards), but auth router logs raw magic-link token + Pear webhook is unauthenticated | 6/10 |
| Performance + DB | **3 blockers / 6 concerns** — connection pool too small + 1 N+1 + 3 missing indexes — all 1-line or 15-min fixes | 7/10 |
| UI/UX + a11y | **4 blockers / 7 concerns** — fake "Maria" hardcoded data + unreachable journey celebration + WCAG AA contrast fail on every muted text | **5.5/10** (Akram-controlled demo 7/10; real members tomorrow **4/10**) |

| Code quality + arch | **3 blockers / 7 concerns** — dual reward model + `CompassCHW-Mobile` shadow repo + relationship-gate fragmentation | 6.5/10 (solo-CTO operable; second-engineer-blocker score) |

**Overall pre-launch confidence: 6/10.** The platform can hold up under cofounder-controlled demo. It cannot hold up under real Medi-Cal members from cold start until the launch checklist below is shipped.

## Detailed findings by dimension

### A. Security & HIPAA

**Pass (verified solid):**
- JWT implementation: HS256 with `jti` on every token, `type` claim validation, refresh tokens stored as SHA-256 hashes only, Argon2 password hashing
- `medi_cal_id` AES-256-GCM column encryption via `EncryptedString`; key from env, never committed
- CORS: explicit origin whitelist, no `*`, `allow_credentials=True` correctly paired
- Services consent gate fires on call-bridge, send-session-message, send-conversation-message, request-accept
- Vonage webhook signature verification (HS256 JWT + body hash + replay window)
- Stripe + Pear webhook signature verification at the boundary
- BAA gates at startup: `sys.exit(1)` in prod if any of {AssemblyAI, Anthropic, Vonage, Pear} BAA env confirmations missing
- No hardcoded secrets in any committed `.py` file

**Beyond the top-10 blockers, additional concerns:**
- **`EncryptedString` decryption-failure fallback returns plaintext silently** (`utils/encryption.py:63-69`). Should log warning + add `phi_encryption_migrated_at` column.
- **`assert_shared_session` accepts ANY status including cancelled-2-years-ago.** No expiration on the care relationship. Implication: ex-CHWs retain access to ex-members' PHI forever.
- **No `phi_read` audit row** on: case-note list, transcript read, session documentation read, member document read. Only the CHW roster endpoint writes this. Required for HIPAA §164.312(b) completeness.
- **`admin_key` is static, no expiry, no rotation log.** If leaked, the only remediation is rotation + redeploy. No idea how long exposure ran.
- **Admin 2FA JWT has no `jti`.** Replayable within 15-min TTL window.
- **Mapbox token in client bundle** — confirm restricted to `joincompasschw.com` + bundle ID in Mapbox dashboard.

### B. Performance + DB

**Pass:**
- Audio finalization pipeline correctly off the request path (background webhook → finalizer → S3 PUT + AssemblyAI + Claude summary). CHW Complete button is fast.
- AI summary generation correctly backgrounded via FastAPI `BackgroundTasks`.
- Existing composite indexes on conversations, messages, audit_log are sound.

**Capacity model (t3.medium + RDS single-AZ):**

| Concurrent users | Resource that tips first | Mitigation |
|---|---|---|
| 50 | Comfortable | n/a |
| 80-100 | DB connection pool exhaustion via `/conversations/` N+1 | Blocker #7 fix raises ceiling to ~150 |
| 150-200 | Audio finalization RAM ceiling (~25 MB × N concurrent) | Move finalizer to dedicated worker |
| 200+ | Single uvicorn workers + APScheduler in-process | Multi-instance + externalize scheduler |

**Beyond top-10, additional concerns:**
- **`PATCH /sessions/{id}/complete` (legacy)** still synchronously awaits Vonage transcript fetch (`routers/sessions.py:435-450`). Vonage's transcript is a polling job — blocks the response for 30-120s. Background it or delete the block (new flow uses `/end` + webhook).
- **3 missing indexes** — addable in one migration with `CREATE INDEX CONCURRENTLY`:
  - `sessions (chw_id, pinned_at DESC NULLS LAST, created_at DESC) WHERE deleted_at IS NULL` — CHW inbox sort
  - `session_transcripts (session_id, started_at_ms)` — summarizer sort
  - `billing_claims (pear_suite_claim_id) WHERE pear_suite_claim_id IS NOT NULL` — scheduler polls
- **`StaggerList` delay accumulation**: 20 cards × 50ms = last card invisible for 1 second. Cap at 300ms total. (One-line fix: `Math.min(index * delayMs, 300)`)
- **AI summary refetch interval bug**: `useApiQueries.ts:1464` polls every 4s indefinitely even after summary arrives. Should stop when `ai_summary != null`.

### C. UI/UX + accessibility

**Pass:**
- Card / Pill / StatTile / PageHeader / EmptyState primitives are clean and well-adopted
- Polish kit (tabular nums, lucide icons, tinted shadows, motion) actually shipped and visible
- MemberCalendar empty state copy is exemplary: "Connect with a Community Health Worker to schedule your first session" + CTA
- Bilingual welcome strip on Member side is the right instinct (typography as identity, not a settings toggle)

**The "Maria" finding deserves elevation:** static hardcoded names in production Recent Activity (`MemberHomeScreen:784-813`) AND in MemberMessages bilingual greeting (uses feminine `"Bienvenida"` always — assumes the user is a woman). Both are demo-killers within 10 seconds of a real session.

**The unreachable celebration:** `JourneyStepSpring` (the only piece of delight engineering in the polish kit — overshoot spring + floating "+pts" chip) is **wired in exactly ONE place** (MemberMessages right rail). The `completed` prop arrives already-true so the `false→true` edge transition never fires. Should run on actual step-complete mutation. The pop never happens.

**Beyond top-10:**
- **`iconBtnCard` touch target 34×34dp** — under WCAG 2.5.5 minimum 44×44. Used on Phone, Calendar, MoreVertical in conversation headers. Mis-tap risk during live calls.
- **No per-field inline validation** on RegisterScreen. Single top banner. CIN format only checks on submit. Member who fills 8 of 9 fields sees greyed-out button with no idea why.
- **Two journey-like sidebar entries** ("My Journey" → MemberJourneyScreen, "Roadmap" → MemberRoadmapScreen) with similar names. Will confuse first-time users.
- **`window.confirm()` for call confirmation on web** (`MemberMessagesScreen:1418`). Browser-blocking, popup-blocker-vulnerable, unstyleable.
- **No first-run onboarding overlay** for fresh CHW account on mobile. Just stat tiles showing zeroes.

### D. Production readiness + SRE

**Pass:**
- Auto-deploy pipeline is reliable since the 2026-06-09 fixes (correct compose service name, PYTHONPATH for alembic, `script_stop: false`)
- Migrations apply cleanly on deploy; single head chain since `ab1c2d3e4f5a` multi-head merge
- 5 PHI buckets with KMS + versioning + 7-year HIPAA lifecycle (runbook works)
- Documented `force-recreate` gotcha for env-var changes
- ADMIN_2FA_SECRET rotated 2026-06-09

**Beyond top-10:**
- **`/api/v1/health` checks env-var presence, not connectivity** (`routers/health.py:93-117`). Rotated Vonage key shows "ok" until first real call 401s. Should ping Vonage `GET /v2/applications/{app_id}` etc.
- **CloudWatch log retention "never expire"** since April 2026. HIPAA §164.312(b) + accumulating storage cost.
- **No tested deploy rollback procedure.** With 5 historical heads + auto-`upgrade heads`, a bad migration requires manual SSM + `alembic downgrade` + `--force-recreate`. Never drilled.
- **No CloudWatch alarms** on RDS CPU/storage or EC2 CPU.
- **No documented Vonage rotation procedure** (still deferred per vault Open Questions).
- **No JWT signing key rotation** (`SECRET_KEY`) ever. No `kid` claim in JWT. Stolen env file = forge any token.
- **CHW deactivation has no member-handoff path.** Karla retires with 5 active members → orphaned silently.

### E. Code quality + architecture

**Pass:**
- Pydantic v2 schemas at every boundary
- Tests exist + run in CI
- Migration chain clean after multi-head merge
- Service layer pattern mostly consistent (relationship guards, audit log, etc.)
- Shared design primitives (`ui/`) adopted across most screens

**Beyond top-10:**
- **`sessions.py` is a 2,198-line god router** with 14 inline `import logging` statements. 8 logical domains (CRUD, consent, messaging, transcript, AI summary, followups, billing claim) entangled. Should split.
- **`useApiQueries.ts` is 3,185 lines** mixing every type, queryKey, query, and mutation. Probably the cause of `tsc --noEmit` stack overflow. Should split per-domain.
- **`Platform.OS` scattered in 55 files** (140 occurrences in screens). Should be in a small set of platform hooks, not at the render layer.
- **TODO stubs in billing-adjacent code** — `WellnessPointsLedger` returns HTTP 503 on write failure but the redemption silently completes anyway. Member can redeem points without points being deducted.
- **CompassCHW-Mobile shadow repo** diverged from `native/`. No CI sync. A second engineer could land in either dir and produce a different binary.
- **Dual reward models** — `reward.py` (old: `reward_transactions` + `redemption_items`) and `rewards.py` (new: `reward_catalog_items` + `reward_redemptions`). Both imported in prod paths. Canonical?
- **`web-legacy/.vercel/project.json`** is live and pointed at `compass-admin`. A `vercel deploy` from that directory would push legacy admin over the live admin project. Footgun.

## Notes for Akram

The audit is calibrated to "what blocks a real first session" — not "what would a senior PE flag in code review." The codebase reflects strong taste applied at speed: the BAA gates, the encryption-at-rest, the relationship guards, the polish kit, the Messages rebuilds, and the HIPAA bucket restructure are all above-average for a one-person platform.

The findings cluster in two real categories:

1. **The "looks polished but breaks on first real user" gap.** The "Maria" hardcoded recent-activity feed, the unreachable journey celebration, the Sentry that isn't installed — these are all places where the system passes a controlled demo but fails when stress hits cold-start reality. These are the cheapest to fix and highest-leverage to address before Karla brings her first member onto Compass.

2. **The "compounds badly with a second engineer" gap.** Dual reward models, the `CompassCHW-Mobile` shadow repo, the 3,185-line useApiQueries.ts — these don't block launch but will burn 30% of a new engineer's onboarding week.

Categories 1+2 are addressable. None require throwing away architecture. Most are ≤4 hour fixes.

Recommendation: ship Sprint 1 + Sprint 2 in 2-3 days as a focused block. That clears the launch-blocker set and pulls overall confidence to 8/10. Sprints 3-4 can ship over the following week. Sprint 5 lines up before any formal HIPAA audit and is the right gate for taking on a second CHW.
