# CHW SMS 2FA Implementation Plan (Spec 2)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Required SMS 2FA for CHW logins (opt-in for members) with 30-day trusted devices, login-time enrollment, and admin recovery.

**Architecture:** Post-password challenge via a single-purpose pending JWT accepted by exactly two new `/auth/2fa/*` endpoints; OTP reuses the `PhoneVerification` machinery + the unified Messages client; trusted devices are hashed tokens with 30-day expiry; one migration.

**Spec (binding):** `docs/specs/2026-07-15-chw-sms-2fa-design.md` — read it fully first, especially Security invariants (every one becomes a test).

## Global Constraints

- Same as Spec 1's plan (`docs/plans/2026-07-15-sms-output.md` Global Constraints): best-effort SMS, no-PHI bodies, masked logging, ≥85% backend diff coverage, bun (not npm) for FE, isolated test DB per agent, foreground test runs, conventional commits ending `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`.
- NOTHING in this plan changes admin login (TOTP flow untouched) or the Spec-1 phone-verification endpoints' contracts.
- The pending JWT is signed with `settings.admin_2fa_secret` but MUST use type claim `user_2fa_pending` (never `admin_2fa`) — cross-acceptance is a security bug; test both directions.

## PR structure

- **PR-A `feat/sms2-2fa-backend`** — Tasks 1–6 (migration, challenge flow, trusted devices, admin reset). Merge FIRST.
- **PR-B `feat/sms2-2fa-frontend`** — Tasks 7–10 (TwoFactorScreen, device-token plumbing, CHW settings card, member toggle, CHW signup routing). Built in parallel against the contract below; rebase on main after PR-A merges, verify, then merge.

**The API contract PR-B builds against** (PR-A implements exactly this):
- `POST /api/v1/auth/login` → EITHER today's token response OR `200 {"two_fa_required": true, "pending_token": str, "phone_verification_required": bool, "phone_last4": str|null}`. Request may carry header `X-Device-Token`.
- `POST /api/v1/auth/2fa/send-code` `{pending_token: str, phone?: str}` → `200 {"sent": true, "phone_last4": str}` | 401 bad/expired pending | 422 sentinel/invalid phone | 409 duplicate phone | 429 rate-limited.
- `POST /api/v1/auth/2fa/verify` `{pending_token: str, code: str, remember_device: bool}` → today's full token response `+ {"device_token": str}` when remember_device | 401 bad pending | 410 expired/exhausted code | 422 wrong code (attempts decrement).

### Task 2 (PR-A): Pending-token service

**Files:** Create `backend/app/services/user_2fa.py`; Test: `backend/tests/test_chw_sms_2fa.py`.

**Produces:**
```python
PENDING_TOKEN_TYPE = "user_2fa_pending"
PENDING_TTL_MINUTES = 10
def issue_pending_token(user_id: UUID) -> str            # jwt.encode({sub,type,exp}, settings.admin_2fa_secret, "HS256")
def decode_pending_token(token: str) -> UUID              # raises HTTPException(401) on bad sig/expiry/wrong type
def hash_device_token(raw: str) -> str                    # hashlib.sha256(raw.encode()).hexdigest()
def mint_device_token() -> str                            # secrets.token_urlsafe(32)
async def user_requires_2fa(user: User) -> bool           # chw: settings.chw_sms_2fa_enabled; member: user.sms_2fa_enabled and phone_verified_at and not is_placeholder_phone(normalized phone); admin/other: False
async def find_valid_trusted_device(db, user_id, raw_token: str|None) -> TrustedDevice|None  # hash match + user match + unexpired
```
TDD each function (invalid signature 401, expired 401, `admin_2fa`-typed token REJECTED, member gating matrix incl. sentinel, device expiry/user-mismatch). Commit `feat(2fa): pending-token + trusted-device service`.

### Task 4 (PR-A): `/auth/2fa/send-code` + `/auth/2fa/verify`

**Files:** Modify `backend/app/routers/auth.py` (two endpoints); extract reusable OTP helpers from `backend/app/routers/phone_verification.py` into `backend/app/services/otp.py` (`create_otp(db, user_id, phone) -> raw_code` with the 3/hour cap, `check_otp(db, user_id, code) -> phone_e164` with attempts/TTL/410 semantics) and refactor phone_verification.py to use them (behavior-identical — its existing tests must pass untouched).

send-code: decode pending → load user → resolve target phone (verified phone; else `phone` param required → normalize, sentinel 422, duplicate pre-check per phone-uniqueness rules 409) → `create_otp` → send via unified client (`brand_outbound_sms`, copy: `"Your Compass sign-in code is {code}. It expires in {TTL} minutes."`) → `{"sent": true, "phone_last4": ...}`; delivery failure → 500, row kept. IP rate limit `@limiter.limit("10/minute")`.
verify: decode pending → `check_otp` → on enrollment path set `User.phone`/`phone_verified_at` (IntegrityError → 409) → issue standard tokens (reuse login's token-issuance code path) → if `remember_device`: mint token, store hash w/ 30-day expiry + request User-Agent, include raw in response.
Tests: full happy path CHW (challenge → send → verify → tokens work on a protected endpoint); enrollment path sets verified phone; wrong code 422 decrements, exhaustion 410; expired pending 401; pending token rejected by any OTHER endpoint (pick `/chw/members` + `/member/profile`) and real access token rejected by 2FA endpoints; remember_device stores hash-only (assert raw not in DB) and next login bypasses; rate limits. Commit `feat(2fa): challenge endpoints with login-time enrollment`.
